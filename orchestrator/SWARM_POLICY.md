# Swarm Policy（Elvis 风格）

本文件定义“OpenClaw 编排层（Zoe）如何管理 Claude Code 执行层”以实现高频稳定交付。

## 核心原则

1. **两层上下文分离**
   - Zoe：业务/客户/历史决策/验收标准（不塞代码库全文）
   - Executor（Claude Code）：代码库/类型/测试/CI（不背业务长历史）

2. **上下文窗口是零和博弈**
   - 业务与代码不可同时塞满：必须拆层。

3. **高频提交的来源**
   - 不是“大任务一次写完”，而是 **微任务工单化**。
   - 默认：**1 微任务 = 1 commit**，10~30 分钟为一个节拍。

## 微任务拆分方法（强约束）

每个需求拆成若干类任务：

- 侦察类：定位入口/类型/依赖/风险
- 设计类：接口/数据结构/迁移方案
- 实现类：每个 agent 只改 1~2 个模块或 1~3 个文件
- 测试类：单测/集成测/回归用例
- 修复类：CI、lint、类型错误
- 文档类：README、变更记录、使用示例

### 微任务判定标准

一个微任务必须满足：
- 可独立验收（有 DoD）
- 影响面可控（明确文件列表）
- 可回滚（说明回滚/feature flag/兼容策略）

## 并行策略

- 并行上限（建议）：3~7 个执行 agent
- 优先并行：侦察/实现/测试
- 避免并行：同一文件的重叠修改（冲突成本高）

## 合并策略

- Executor 输出 commit（Conventional Commits）
- Zoe 聚合为 PR：
  - Summary（做了什么）
  - Files changed（改了哪些）
  - Tests（怎么测）
  - Risks & rollback（风险与回滚）

## 失败处理

- 失败只允许两种动作：
  1) 缩小任务范围
  2) 增加侦察/测试

不要“强行继续堆上下文”。
